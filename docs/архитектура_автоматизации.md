## Назначение файлов и папок

| Путь | Что хранит |
|---|---|
| `package.json` | Node-проект, зависимости и команды запуска: `login`, `compare-cards`, `niche-report` |
| `playwright.config.ts` | Настройки Playwright: браузер, таймауты, viewport, режим запуска |
| `config/scenario.json` | Параметры сценария: категория, предмет, период, способ отбора топа |
| `config/runtime.json` | Технические параметры запуска: задержки между шагами, таймауты, viewport |
| `src/cli/login.ts` | Ручной вход в WB Partners и сохранение авторизованной сессии |
| `src/cli/run-niche-report.ts` | Главная точка запуска основного сценария сбора отчета ниши |
| `src/cli/run-compare-cards.ts` | Точка запуска сценария `Сравнение карточек` |
| `src/core/browser.ts` | Запуск браузера, подключение сохраненной сессии, закрытие браузера |
| `src/core/config.ts` | Чтение `scenario.json` и `runtime.json` |
| `src/core/pacing.ts` | Human-like pacing: мягкие случайные задержки перед действиями и между шагами |
| `src/core/stepRunner.ts` | Обертка над шагами: старт, успех, ошибка, длительность, номер шага |
| `src/core/logger.ts` | Текстовые логи для консоли и Airflow |
| `src/core/incidents.ts` | Классификация ошибок в `incident_type` |
| `src/flows/nicheReportFlow.ts` | Основной сценарий: открыть прямой URL отчета, распарсить страницу, сохранить в БД |
| `src/flows/nicheUiFallbackFlow.ts` | Fallback-сценарий: найти нишу через UI-фильтры, если прямой URL не сработал |
| `src/flows/compareCardsFlow.ts` | Отдельный сценарий `Сравнение карточек`: открыть страницу сравнения, выбрать рекомендации по предмету, топ карточек, собрать 50 ID, сохранить ID и добавить первые 5 карточек через ручной ввод |
| `src/steps/` | Отдельные Playwright-функции из гранулярности сценария |
| `logs/` | Локальные текстовые или JSON-логи запусков |
| `airflow/` | DAG Airflow, который запускает основной сценарий |
| `docs/` | Документация по сценарию и архитектуре |

## Главная логика

Airflow не управляет отдельными кликами. Airflow запускает один крупный task:

```text
run_niche_report_flow
```

Основной путь открывает готовую ссылку отчета:

```text
https://seller.wildberries.ru/platform-analytics/niche-analysis/item?id=643
```

`id=643` — идентификатор ниши/предмета WB. Если URL не открыл нужную страницу, Playwright запускает fallback через UI-фильтры.

Пример логов основного пути:

```text
[1/4] openNicheReportByUrl success
[2/4] setNichePeriodMonth success
[3/4] parseNicheReport success
[4/4] saveNicheReportToDb success
```

Пример fallback-логов для открытия отчета ниши через UI-фильтры:

```text
[01/09] openAnalyticsNichePage success
[02/09] setPeriodMonth success
[03/09] openFilters success
[04/09] resetFiltersIfActive success
[05/09] selectCategory success
[06/09] selectSubject success
[07/09] applyFilters success
[08/09] openNicheCard success
[09/09] setNichePeriodMonth success
```

Отдельный сценарий `Сравнение карточек` начинается со своей нумерации.

```text
[01/08] openCompareCardsPage success
[02/08] startCompareCards success
[03/08] selectRecommendationsBySubject success
[04/08] searchAndSelectCompareSubject success
[05/08] selectTopByRevenue success
[06/08] parseCompareCardIds success
[07/08] saveCompareCardIdsToDb success
[08/08] addManualCompareCards success
```

Если сценарий падает:

- Airflow показывает, что упал task `run_compare_cards_flow`;
- внутренний step-log показывает, на каком именно шаге и почему;
- `incident_type` показывает класс проблемы: `auth_expired`, `captcha`, `selector_changed`, `popup_blocking`, `timeout`, `business_limit`, `empty_result`, `invalid_niche_url`, `schema_changed`, `unknown_screen`.

## Поток запуска

```text
login.ts
  -> открывает WB Partners
  -> менеджер вручную входит в аккаунт
  -> сохраняется сессия

run-niche-report.ts
  -> читает config/scenario.json
  -> читает config/runtime.json
  -> запускает browser.ts
  -> запускает nicheReportFlow.ts
  -> сначала пробует открыть scenario.nicheReportUrl
  -> если direct URL не сработал, запускает nicheUiFallbackFlow.ts как fallback
  -> parseNicheReport.ts собирает данные со страницы
     -> сохраняет заголовок блока сезонности как metric_code=seasonality_title
     -> в блоке "Динамика по предмету" переключает 3 режима и собирает только агрегированные значения над графиком
     -> перед парсингом поисковых запросов скроллит таблицу и нажимает "Показать ещё" до 50 строк
     -> динамика WB сохраняется со знаком: красная стрелка вниз = отрицательный delta_value
     -> динамика конверсий в поисковых запросах тоже сохраняется со знаком и direction
  -> saveNicheReportToDb.ts сохраняет результат в PostgreSQL
  -> каждый шаг проходит через stepRunner.ts
  -> pacing.ts добавляет мягкие задержки между шагами
  -> logger.ts пишет логи
  -> incidents.ts классифицирует ошибку, если сценарий упал
```

## Runtime-безопасность

Автоматизация не кликает по координатам и не использует фиксированные долгие `sleep`.

Правило:

```text
шаг делает действие -> шаг ждет конкретный DOM/result -> stepRunner делает мягкую задержку перед следующим шагом
```

Технические задержки живут в `config/runtime.json`:

```json
{
  "delays": {
    "beforeActionMs": [300, 700],
    "betweenStepsMs": [800, 1500],
    "afterNavigationMs": [2000, 4000],
    "typingDelayMs": [30, 80]
  }
}
```

`pacing.ts` отвечает только за темп выполнения. Проверки успеха остаются внутри конкретных `steps/*.ts`.

## Сохранение данных

Основной результат пишется в локальную PostgreSQL БД на `127.0.0.1:7777`.
